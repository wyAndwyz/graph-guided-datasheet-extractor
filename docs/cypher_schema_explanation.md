# Cypher Schema Explanation

This document explains the Cypher schema files used to initialize the Neo4j graph for the graph-guided datasheet extraction demo.

The goal is to make the graph initialization process understandable, auditable, and easy to extend.

The schema files are located in:

```text
graph/schema/
├─ 01_create_constraints.cypher
├─ 02_init_control_graph.cypher
├─ 03_equipment_schema.cypher
├─ 04_parameter_schema.cypher
└─ 05_unit_vocabulary.cypher
```

## Overview

The Neo4j graph is initialized in two layers:

1. **Instance graph layer**  
   This stores extracted information from datasheets, such as equipment, technical parameters, units, evidence, extraction runs, and validation issues.

2. **Control/vocabulary graph layer**  
   This stores semantic vocabulary and lightweight constraints used to guide document screening, LLM extraction, vocabulary grounding, and validation.

The schema files do not yet load extracted datasheet information. They only define constraints, controlled vocabulary, relationship definitions, and parameter metadata.

---

# 1. `01_create_constraints.cypher`

## Purpose

This file creates uniqueness constraints for the main node labels.

Constraints ensure that important nodes can be safely created or merged without duplication.

For example, if every `Datasheet` node has a unique `id`, the graph loader can safely use:

```cypher
MERGE (d:Datasheet {id: $id})
```

instead of accidentally creating duplicate datasheet nodes.

## Example Statement

```cypher
CREATE CONSTRAINT datasheet_id_unique IF NOT EXISTS
FOR (d:Datasheet)
REQUIRE d.id IS UNIQUE;
```

## Syntax Explanation

### `CREATE CONSTRAINT`

Creates a database constraint.

A constraint is a database-level rule that Neo4j enforces automatically.

### `datasheet_id_unique`

This is the name of the constraint.

Giving each constraint a clear name makes it easier to inspect, debug, or drop later.

### `IF NOT EXISTS`

This prevents errors if the same constraint has already been created.

This is useful because schema scripts may be executed multiple times during development.

### `FOR (d:Datasheet)`

This means the constraint applies to all nodes with the label `Datasheet`.

Here, `d` is a variable name used inside the statement.

### `REQUIRE d.id IS UNIQUE`

This requires the `id` property to be unique among all `Datasheet` nodes.

No two `Datasheet` nodes may have the same `id`.

## General Pattern

Most constraints in this file follow one of these patterns:

```cypher
CREATE CONSTRAINT <constraint_name> IF NOT EXISTS
FOR (n:<Label>)
REQUIRE n.id IS UNIQUE;
```

or:

```cypher
CREATE CONSTRAINT <constraint_name> IF NOT EXISTS
FOR (n:<Label>)
REQUIRE n.<property> IS UNIQUE;
```

## Instance Graph Constraints

The instance graph constraints use `id` as the stable key:

```text
Datasheet.id
Equipment.id
TechnicalParameter.id
Evidence.id
ExtractionRun.id
ValidationIssue.id
```

These nodes are usually created from graph-shaped JSON generated by the extraction pipeline.

## Control/Vocabulary Graph Constraints

The control/vocabulary graph uses stable vocabulary keys:

```text
Concept.name
ParameterType.name
Unit.symbol
RequiredParameter.id
AllowedUnit.id
RelationshipDefinition.name
```

For example:

```cypher
CREATE CONSTRAINT parameter_type_name_unique IF NOT EXISTS
FOR (pt:ParameterType)
REQUIRE pt.name IS UNIQUE;
```

This means that the vocabulary concept `PeakSensitivityWavelength` should only exist once.

## Design Reason

The constraints support reliable use of `MERGE`.

Without uniqueness constraints, graph loading can silently create duplicate vocabulary nodes, especially when scripts are executed repeatedly.

# 2. `02_init_control_graph.cypher`

## Purpose

This file initializes the lightweight control/vocabulary graph.

It creates:

- core `Concept` nodes;
- `RelationshipDefinition` nodes;
- relationship definitions between concepts;
- initial `ParameterType` vocabulary;
- initial `RequiredParameter` nodes.

This file defines the semantic backbone of the MVP graph.

---

## 2.1 Core Concepts

Example:

```cypher
MERGE (c:Concept {name: "Datasheet"})
SET
  c.description = "A technical datasheet document used as input for extraction.",
  c.category = "instance_label";
```

## Syntax Explanation

### `MERGE`

`MERGE` finds an existing pattern or creates it if it does not exist.

Here:

```cypher
MERGE (c:Concept {name: "Datasheet"})
```

means:

> Find a `Concept` node with `name = "Datasheet"`.  
> If it does not exist, create it.

This is different from `CREATE`, which always creates a new node.

### `(c:Concept {name: "Datasheet"})`

This describes a node pattern.

- `c` is the variable name.
- `Concept` is the node label.
- `{name: "Datasheet"}` is a property map.

### `SET`

`SET` assigns or updates properties on the matched or created node.

For example:

```cypher
SET
  c.description = "...",
  c.category = "instance_label";
```

adds or updates two properties:

```text
description
category
```

### Semicolon `;`

The semicolon ends one Cypher statement.

A variable such as `c` does not persist across separate statements.

This is important: if a later statement needs a node again, it must use `MATCH` or `MERGE` again.

## Created Core Concepts

The MVP creates these core concepts:

```text
Datasheet
Equipment
TechnicalParameter
ParameterType
Unit
Evidence
ExtractionRun
ValidationIssue
```

The `category` property distinguishes whether a concept represents an instance label or vocabulary label.

Example:

```text
Datasheet          instance_label
Equipment          instance_label
ParameterType      vocabulary_label
Unit               vocabulary_label
```

---

## 2.2 Relationship Definitions

Example:

```cypher
MERGE (rd:RelationshipDefinition {name: "HAS_PARAMETER"})
SET rd.description = "Links equipment to an extracted technical parameter.";
```

## Explanation

A `RelationshipDefinition` node describes an allowed relationship type in the graph.

For example, the actual instance graph may contain:

```cypher
(:Equipment)-[:HAS_PARAMETER]->(:TechnicalParameter)
```

The control graph represents the meaning of that relationship using a node:

```cypher
(:RelationshipDefinition {name: "HAS_PARAMETER"})
```

This makes the relationship vocabulary queryable and usable for prompt construction or validation.

## Created Relationship Definitions

The initial relationship definitions are:

```text
DESCRIBES
HAS_PARAMETER
HAS_PARAMETER_TYPE
HAS_UNIT
EVIDENCED_BY
GENERATED_BY
ABOUT
```

Later, `03_equipment_schema.cypher` adds:

```text
HAS_EQUIPMENT_TYPE
```

---

## 2.3 Relationship Domains and Ranges

Example:

```cypher
MATCH (fromConcept:Concept {name: "Equipment"})
MATCH (toConcept:Concept {name: "TechnicalParameter"})
MATCH (rd:RelationshipDefinition {name: "HAS_PARAMETER"})
MERGE (rd)-[:FROM_CONCEPT]->(fromConcept)
MERGE (rd)-[:TO_CONCEPT]->(toConcept);
```

## Syntax Explanation

### `MATCH`

`MATCH` finds existing nodes or patterns.

Unlike `MERGE`, it does not create missing nodes.

Here:

```cypher
MATCH (fromConcept:Concept {name: "Equipment"})
```

finds the existing `Concept` node for `Equipment`.

### Multiple `MATCH` statements

The query finds three existing nodes:

```text
Concept: Equipment
Concept: TechnicalParameter
RelationshipDefinition: HAS_PARAMETER
```

Then it creates two control relationships:

```cypher
(rd)-[:FROM_CONCEPT]->(fromConcept)
(rd)-[:TO_CONCEPT]->(toConcept)
```

### Why this is needed

It records that:

```text
HAS_PARAMETER
from: Equipment
to: TechnicalParameter
```

This information can later be retrieved to guide LLM output structure.

## Important Cypher Rule

Variables do not survive after a semicolon.

Therefore this is unsafe:

```cypher
MERGE (a:Concept {name: "A"});
MERGE (b:Concept {name: "B"});
MERGE (a)-[:REL]->(b);
```

The final statement does not know what `a` and `b` are.

The safe pattern is:

```cypher
MATCH (a:Concept {name: "A"})
MATCH (b:Concept {name: "B"})
MERGE (a)-[:REL]->(b);
```

This is why the corrected schema files use `MATCH` before relationship creation.

---

## 2.4 ParameterType Vocabulary

Example:

```cypher
MERGE (pt:ParameterType {name: "PeakSensitivityWavelength"})
SET
  pt.description = "The wavelength at which the sensor has peak sensitivity.",
  pt.value_type = "number",
  pt.priority = "required";
```

## Explanation

`ParameterType` nodes are controlled vocabulary entries.

They represent normalized semantic meanings of extracted parameters.

For example, a datasheet may write:

```text
peak sensitivity
peak wavelength
λp
```

The graph can normalize these to:

```text
PeakSensitivityWavelength
```

## Properties

### `description`

Human-readable explanation of the parameter type.

### `value_type`

Expected data form, such as:

```text
string
number
number_or_range
string_or_dimension_tuple
```

### `priority`

Indicates whether the parameter is required or optional for the MVP.

Example values:

```text
required
optional
```

## Initial Parameter Types

The MVP includes:

```text
Manufacturer
ProductModel
DeviceType
PackageType
PackageForm
Dimensions
PeakSensitivityWavelength
AngleOfHalfSensitivity
```

---

## 2.5 Connecting ParameterType to Concept

Example:

```cypher
MATCH (parameterTypeConcept:Concept {name: "ParameterType"})
MATCH (pt:ParameterType {name: "ProductModel"})
MERGE (pt)-[:INSTANCE_OF_CONCEPT]->(parameterTypeConcept);
```

## Explanation

This connects a concrete vocabulary entry to the general concept category.

It means:

```text
ProductModel is an instance of the ParameterType concept.
```

This is useful because the graph can retrieve all parameter type vocabulary entries with:

```cypher
MATCH (pt:ParameterType)-[:INSTANCE_OF_CONCEPT]->(:Concept {name: "ParameterType"})
RETURN pt.name;
```

---

## 2.6 RequiredParameter Nodes

Example:

```cypher
MERGE (rp:RequiredParameter {id: "required_product_model"})
SET rp.reason = "The product model identifies the device described by the datasheet.";
```

## Explanation

A `RequiredParameter` node represents an extraction target that should exist for the MVP.

It is then connected to the corresponding `ParameterType`:

```cypher
MATCH (rp:RequiredParameter {id: "required_product_model"})
MATCH (pt:ParameterType {name: "ProductModel"})
MERGE (rp)-[:REQUIRES_PARAMETER_TYPE]->(pt);
```

This means:

```text
The MVP requires a parameter of type ProductModel.
```

## Why not just use `priority = required`?

The `priority` property is useful for quick filtering.

The `RequiredParameter` node is more explicit and can later store:

```text
reason
scope
severity
validation rule
applicability
```

This is more flexible for validation.

# 3. `03_equipment_schema.cypher`

## Purpose

This file initializes equipment-related vocabulary.

It introduces:

- `EquipmentType` vocabulary entries;
- aliases for equipment-type grounding;
- the `HAS_EQUIPMENT_TYPE` relationship definition.

This supports equipment type normalization.

---

## 3.1 Relationship Definition: `HAS_EQUIPMENT_TYPE`

Example:

```cypher
MERGE (rd:RelationshipDefinition {name: "HAS_EQUIPMENT_TYPE"})
SET rd.description = "Links extracted equipment to its normalized equipment type.";
```

This adds a relationship definition for:

```cypher
(:Equipment)-[:HAS_EQUIPMENT_TYPE]->(:EquipmentType)
```

The domain and range are represented as:

```cypher
MATCH (fromConcept:Concept {name: "Equipment"})
MATCH (toConcept:Concept {name: "Equipment"})
MATCH (rd:RelationshipDefinition {name: "HAS_EQUIPMENT_TYPE"})
MERGE (rd)-[:FROM_CONCEPT]->(fromConcept)
MERGE (rd)-[:TO_CONCEPT]->(toConcept);
```

## Design Note

In the MVP, `EquipmentType` is treated as a vocabulary specialization under the broader `Equipment` concept.

Therefore both `FROM_CONCEPT` and `TO_CONCEPT` are connected to `Equipment`.

A later version may introduce a separate `Concept {name: "EquipmentType"}` if a more detailed schema is needed.

---

## 3.2 EquipmentType Vocabulary

Example:

```cypher
MERGE (et:EquipmentType {name: "AmbientLightSensor"})
SET
  et.display_name = "Ambient light sensor",
  et.description = "A sensor device that detects ambient light intensity.",
  et.scope = "mvp",
  et.source_product = "Vishay TEMT6000X01";
```

## Explanation

`EquipmentType` nodes normalize raw device-type descriptions.

For example:

```text
ambient light sensor
light sensor
photosensor
```

can all be grounded to:

```text
AmbientLightSensor
```

The MVP includes:

```text
AmbientLightSensor
DigitalAccelerometer
```

`AmbientLightSensor` is used for the Vishay TEMT6000X01 MVP datasheet.

`DigitalAccelerometer` is included for the later Analog Devices ADXL345 variation case.

---

## 3.3 Aliases

Example:

```cypher
MATCH (et:EquipmentType {name: "AmbientLightSensor"})
SET et.aliases = [
  "ambient light sensor",
  "ambient light sensors",
  "light sensor",
  "photo sensor",
  "photosensor"
];
```

## Explanation

Aliases are alternative surface forms that may appear in datasheets.

They can be used by:

- document screening;
- prompt construction;
- vocabulary matching;
- LLM grounding.

---

# 4. `04_parameter_schema.cypher`

## Purpose

This file enriches `ParameterType` nodes with metadata used for screening, extraction, grounding, and validation.

The file does not create new parameter types.  
It adds detailed metadata to the `ParameterType` nodes created in `02_init_control_graph.cypher`.

---

## Example Statement

```cypher
MATCH (pt:ParameterType {name: "Dimensions"})
SET
  pt.display_name = "Dimensions",
  pt.parameter_category = "physical_dimension",
  pt.expects_unit = true,
  pt.screening_weight = 4,
  pt.extraction_hint = "Extract the physical dimensions of the device. Preserve the dimensional expression and unit, for example 4 x 2 x 1.05 mm.",
  pt.aliases = [
    "dimensions",
    "dimension",
    "size",
    "length",
    "width",
    "height",
    "L x W x H"
  ],
  pt.screening_keywords = [
    "dimensions",
    "dimension",
    "size",
    "length",
    "width",
    "height",
    "mm"
  ];
```

## Syntax Explanation

### `MATCH`

Finds the existing `ParameterType` node.

This assumes the node has already been created in `02_init_control_graph.cypher`.

### `SET`

Adds or updates multiple properties.

### Lists

Cypher supports list properties, for example:

```cypher
pt.aliases = [
  "dimensions",
  "dimension",
  "size"
]
```

These are stored as an array-like property in Neo4j.

## Key Metadata Fields

### `display_name`

Human-readable name for documentation and reports.

### `parameter_category`

Broad category such as:

```text
identification
packaging
physical_dimension
optical
```

### `expects_unit`

Boolean value indicating whether extracted values of this parameter type should normally have a unit.

Examples:

```text
Dimensions                    true
PeakSensitivityWavelength     true
Manufacturer                  false
ProductModel                  false
```

This is useful for validation.

### `screening_weight`

Numeric weight used in document screening.

A parameter with higher `screening_weight` contributes more to page relevance scoring.

### `extraction_hint`

Instructional text that can be included in graph-guided prompts.

### `aliases`

Surface forms that may refer to the same parameter type.

These help vocabulary grounding.

### `screening_keywords`

Keywords used for local document screening before LLM extraction.

# 5. `05_unit_vocabulary.cypher`

## Purpose

This file initializes the unit vocabulary and connects allowed units to parameter types.

It supports:

- document screening;
- prompt construction;
- vocabulary grounding;
- unit validation.

---

## 5.1 Unit Vocabulary

Example:

```cypher
MERGE (u:Unit {symbol: "nm"})
SET
  u.name = "nanometre",
  u.dimension = "length",
  u.scope = "mvp",
  u.description = "Nanometre, used for optical wavelength.";
```

## Explanation

A `Unit` node represents a normalized unit concept.

The stable key is:

```text
Unit.symbol
```

Examples:

```text
mm
nm
degree
V
µA
°C
```

## Properties

### `name`

Full unit name.

### `dimension`

Physical or semantic dimension, such as:

```text
length
angle
voltage
temperature
```

### `scope`

Indicates whether the unit is needed for the MVP or for later variation cases.

Example:

```text
mvp
variation
```

### `description`

Human-readable explanation.

---

## 5.2 Connecting Units to the Unit Concept

Example:

```cypher
MATCH (unitConcept:Concept {name: "Unit"})
MATCH (u:Unit {symbol: "nm"})
MERGE (u)-[:INSTANCE_OF_CONCEPT]->(unitConcept);
```

This means:

```text
nm is an instance of the Unit concept.
```

---

## 5.3 Allowed Units

Example:

```cypher
MATCH (pt:ParameterType {name: "PeakSensitivityWavelength"})
MATCH (u:Unit {symbol: "nm"})
MERGE (pt)-[:ALLOWS_UNIT]->(u);
```

This records that:

```text
PeakSensitivityWavelength allows unit nm.
```

This is later used by unit validation.

For the MVP:

```text
Dimensions                  → mm
PeakSensitivityWavelength   → nm
AngleOfHalfSensitivity      → degree
```

String-valued or categorical parameters such as `Manufacturer`, `ProductModel`, `DeviceType`, `PackageType`, and `PackageForm` intentionally do not have units.

# 6. Useful Inspection Queries

After executing all schema files, the following Cypher queries can be used to inspect the graph.

## Count Nodes by Label

```cypher
MATCH (n)
RETURN labels(n) AS labels, count(n) AS count
ORDER BY labels;
```

## Count Relationships by Type

```cypher
MATCH ()-[r]->()
RETURN type(r) AS relationship_type, count(r) AS count
ORDER BY relationship_type;
```

## Inspect Parameter Types

```cypher
MATCH (pt:ParameterType)
RETURN
  pt.name AS parameter_type,
  pt.value_type AS value_type,
  pt.priority AS priority,
  pt.expects_unit AS expects_unit
ORDER BY parameter_type;
```

## Inspect Unit Vocabulary

```cypher
MATCH (u:Unit)
RETURN
  u.symbol AS symbol,
  u.name AS name,
  u.dimension AS dimension,
  u.scope AS scope
ORDER BY scope, symbol;
```

## Inspect Allowed Units

```cypher
MATCH (pt:ParameterType)-[:ALLOWS_UNIT]->(u:Unit)
RETURN
  pt.name AS parameter_type,
  collect(u.symbol) AS allowed_units
ORDER BY parameter_type;
```

## Inspect Required Parameters

```cypher
MATCH (rp:RequiredParameter)-[:REQUIRES_PARAMETER_TYPE]->(pt:ParameterType)
RETURN
  rp.id AS required_parameter,
  pt.name AS parameter_type,
  rp.reason AS reason
ORDER BY parameter_type;
```

## Inspect Relationship Definitions

```cypher
MATCH (rd:RelationshipDefinition)-[:FROM_CONCEPT]->(from:Concept),
      (rd)-[:TO_CONCEPT]->(to:Concept)
RETURN
  rd.name AS relationship,
  from.name AS from_concept,
  to.name AS to_concept
ORDER BY relationship;
```

---

# 7. Summary

The five schema files create a lightweight Neo4j control graph for graph-guided datasheet extraction.

The schema supports:

- stable node creation through constraints;
- controlled parameter and unit vocabulary;
- equipment type grounding;
- relationship definitions;
- required-parameter specification;
- local document screening before LLM extraction;
- graph-guided prompt construction;
- evidence-grounded graph population;
- later Cypher validation and GraphRAG extension.

The MVP graph is intentionally lightweight. It is designed to be understandable, testable, and extendable rather than a complete ontology.
